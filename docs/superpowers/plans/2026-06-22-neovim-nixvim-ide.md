# Neovim IDE via nixvim — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Turn the minimal `software.neovim` home-manager module into a full, declarative IDE (LSP, tree-sitter, sidebar, completion, telescope, One Dark theme, plus must-have plugins) built on nixvim, affecting only the two darwin home configs.

**Architecture:** Add `nixvim` as a flake input and import its home-manager module into both darwin `homeConfigurations`. Rewrite `home/software/neovim/default.nix` from `programs.neovim` to `programs.nixvim`, keeping the existing `software.neovim.enable` gate. The colorscheme follows each machine's `theme.name` via a new `neovim` block in the shared theme table. LSP is hybrid: ambient servers bundled, project-coupled servers resolved from each project's devShell `PATH`.

**Tech Stack:** Nix flakes, home-manager (release-26.05), nixvim (nixos-26.05), Neovim, navarasu/onedark.nvim, blink.cmp, telescope, neo-tree.

**Testing note (read first):** This is declarative Nix config — there are no unit tests to write first. The real per-task verification is: `nix flake check 'path:.' --all-systems` (catches option/type errors across the flake) **and** building the current machine's home activation package, which fully evaluates the nixvim config and downloads/builds every plugin, grammar, and LSP server:

```
nix build --no-link .#homeConfigurations."o__ni@Stepans-MacBook-Pro".activationPackage
```

A successful build is a strong gate (any bad option name, missing package, or type error fails it). Runtime behavior (colorscheme, LSP attach, keymaps) is smoke-tested headlessly in the final task. Where a nixvim option name is uncertain, the step says to verify it against the pinned branch's docs (`https://nix-community.github.io/nixvim/`) — that verification is part of the step, not a placeholder.

## Global Constraints

- Branch: work happens on `feat/neovim-nixvim-ide` (already created, in place). All commits land there.
- Inputs pin to the 26.05 release line: `nixvim` → `github:nix-community/nixvim/nixos-26.05`, `inputs.nixpkgs.follows = "nixpkgs"`. If that branch does not exist, pin a revision compatible with nixpkgs 26.05.
- Affect **only** the two darwin home configs (`o__ni@Stepans-MacBook-Pro`, `o__ni@DodoBook.local`). Do not modify any `nixosConfigurations` or `users/o__ni`.
- Colorscheme follows `config.theme.name` — never hardcode a style; read `config.theme.colors.neovim`.
- LSP hybrid: `nixd`, `lua_ls`, `html`, `cssls` bundled; `ts_ls`, `pyright` registered with `package = null` (devShell PATH).
- Autocompletion uses `blink-cmp`, not `nvim-cmp`.
- No attribution in commits: no `Co-Authored-By`, no "Generated with" markers. Use the existing git identity.
- Stage files explicitly by path (no `git add -A`). Commit messages: conventional style with `neovim` scope (matches repo history, e.g. `feat(neovim): ...`).
- Format every `.nix` file touched with `nixfmt` before committing (`make fmt` formats all).

```nix
  lib,
  config,
  ...

let
  cfg = config.software.neovim;
  nvimTheme = config.theme.colors.neovim;
in
{
  options.software.neovim = {
    enable = lib.mkEnableOption "neovim editor";
  };

      enable = true;
      defaultEditor = true;

        number = true;
        tabstop = 4;

        shiftwidth = 4;

      '';

      colorschemes.onedark = {
        enable = true;
        settings.style = nvimTheme.style;
      };
    };
  };
}
```

### Task 2: LSP (hybrid) + direnv + formatting

Add the language servers, devShell integration, and format-on-save. After this task, opening a `.nix` or `.lua` file attaches a server; `.ts`/`.py` attach only inside a project devShell.

**Files:**
- Modify: `home/software/neovim/default.nix`

**Interfaces:**
- Consumes: `programs.nixvim` baseline from Task 1.
- Produces: `programs.nixvim.plugins.lsp` with the hybrid server set; `plugins.conform-nvim`; `plugins.direnv`. Later tasks add sibling keys under `programs.nixvim.plugins`.

- [ ] **Step 1: Add the `plugins` block with LSP, direnv, and formatting**

In `home/software/neovim/default.nix`, inside `programs.nixvim`, add a `plugins` attrset after the `colorschemes` block:

```nix
plugins = {
  lsp = {
    enable = true;
    servers = {
      # Ambient — bundled, no project deps
      nixd.enable = true;
      lua_ls.enable = true;
      html.enable = true;
      cssls.enable = true;

      # Project-coupled — resolved from the project's devShell PATH
      ts_ls = {
        enable = true;
        package = null;
      };
      pyright = {
        enable = true;
        package = null;
      };
    };
  };

  # Load a project's .envrc inside nvim so devShell servers land on PATH
  direnv.enable = true;

  # Format on save
  conform-nvim = {
    enable = true;
    settings = {
      format_on_save = {
        timeout_ms = 500;
        lsp_format = "fallback";
      };
      formatters_by_ft = {
        nix = [ "nixfmt" ];
        lua = [ "stylua" ];
        javascript = [ "prettierd" ];
        typescript = [ "prettierd" ];
        typescriptreact = [ "prettierd" ];
        html = [ "prettierd" ];
        css = [ "prettierd" ];
        python = [ "ruff_format" ];
      };
    };
  };
};
```

Verify option names against the pinned branch if the build fails: the LSP helper is `plugins.lsp` with `servers.<name>.enable` / `.package`; the formatter module is `plugins.conform-nvim`; the direnv module is `plugins.direnv`. If `plugins.direnv` is absent in this branch, use `extraPlugins = [ pkgs.vimPlugins.direnv-vim ];` (add `pkgs` to the module args).

- [ ] **Step 2: Format**

Run: `make fmt`
Expected: clean.

- [ ] **Step 3: Build the home activation package**

Expected: builds. Pulls `nixd`, `lua-language-server`, `vscode-langservers-extracted` (html/css), `nvim-lspconfig`, `conform.nvim`, `nixfmt`, `stylua`, `prettierd`, `ruff`. `ts_ls`/`pyright` are **not** built (package = null) — confirm no `typescript-language-server`/`pyright` derivation appears in the build log.
